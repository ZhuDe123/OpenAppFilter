# 技能编写最佳实践

## 核心原则

### 简洁是关键

上下文窗口是公共资源。技能与系统提示、对话历史、其他技能的元数据共享空间。

**默认假设**：AI 已经很聪明。只添加它还不知道的上下文。对每条信息质疑：

- "AI 真的需要这个解释吗？"
- "我能假设 AI 知道这个吗？"
- "这段话值得它的 token 成本吗？"

### 设置适当的自由度

将具体程度与任务的脆弱性和可变性相匹配：

| 自由度 | 适用场景 | 示例 |
|--------|---------|------|
| **高** | 多种方案都有效，决策取决于上下文 | 代码审查流程：给出原则，让 AI 自行判断 |
| **中** | 存在首选模式，允许一些变化 | 提供伪代码模板，允许按需调整 |
| **低** | 操作脆弱易出错，一致性至关重要 | 数据库迁移：提供精确命令，禁止修改 |

### 用实际使用的模型测试

技能效果取决于底层模型。在不同模型上测试以确保兼容。

## 技能结构

### 命名约定

使用一致的命名模式。推荐 kebab-case 小写英文名称（如 `writing-skills`）。

**好的示例**：`writing-skills`、`test-driven-development`、`systematic-debugging`
**避免**：模糊名称如 `helper`、`utils`、`tools`

### 编写有效的描述

`description` 用于技能发现，必须同时包含**做什么**和**何时使用**。

- 用第三人称写（描述注入系统提示，人称不一致会出问题）
- 包含关键术语和触发条件
- 足够具体让系统知道何时选择此技能

好的示例：
```
description: 遇到任何 bug、测试失败或异常行为时使用，在提出修复方案之前执行
```

### 渐进式披露模式

SKILL.md 作为概述，按需指向详细材料。

- SKILL.md 正文保持在 500 行以内
- 接近此限制时将内容拆分到独立文件
- 引用保持一层深度（所有引用文件直接从 SKILL.md 链接）

**目录结构模式**：

- `skill-name/`
  - `SKILL.md` — 主指令（触发时读取）
  - `reference.md` — 参考文件（按需读取）
  - `scripts/` — 可执行脚本目录
    - `helper.py` — 可执行脚本（执行，不读取）

**三种组织模式**：
1. **高层指南加引用**：SKILL.md 给概览，链接到独立文件获取细节
2. **领域特定组织**：按领域分文件，避免加载不相关上下文
3. **条件性细节**：展示基本内容，链接到高级内容

## 工作流和反馈循环

### 对复杂任务使用工作流

将复杂操作分解为清晰的顺序步骤，提供可勾选的清单。

```
任务进度：
- [ ] 步骤 1：描述步骤
- [ ] 步骤 2：描述步骤
- [ ] 步骤 3：描述步骤
```

### 实现反馈循环

运行验证 → 修复错误 → 重复。此模式大幅提高输出质量。

```
1. 执行操作
2. 立即验证（运行验证脚本或对照检查清单）
3. 如果失败：审查错误、修复、再次验证
4. 只有验证通过才继续
```

## 内容指南

### 避免时间敏感信息

使用"旧模式"章节折叠已废弃内容，而非删除。

### 使用一致的术语

选择一个术语并在整个技能中统一使用。混用术语让 AI 难以理解指令。

### 提供示例

示例比纯描述更能帮助 AI 理解期望风格和详细程度。提供输入/输出对。

## 要避免的反模式

- **提供太多选项**：除非必要，不要展示多种方案。提供一个默认方案加一个备用方案
- **深层嵌套引用**：所有引用保持一层深度，确保 AI 能完整读取
- **假设工具已安装**：明确列出所需依赖和安装方式
- **Windows 风格路径**：始终使用正斜杠
